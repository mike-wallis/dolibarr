Core Patch — Supplier Price VAT Rate Dropdown & Code Handling

Decision/patch record for the South Side Supplies / Bright Cleaning Solutions Dolibarr evaluation. Dolibarr version: 23.0.3. File: htdocs/product/price_suppliers.php (core, not tracked in git — see Upgrade risk).

This is the same underlying patch covered day-to-day in the Dolibarr Knowledge Base entry KM2607 — "GST not auto-populating on purchase order lines" (Setup → Knowledge Management). KM2607 explains the symptom/fix from a user's point of view and includes a one-off SQL fix for supplier prices imported before this patch existed. This page is the upgrade-risk record for the code side — keep both in sync if either changes.

The problem

On a product's "For purchase" tab, the "Tax Rate (for this vendor/product)" field (VATRateForSupplierProduct) has two related problems in stock Dolibarr 23.0.3:

  1. No dropdown. The proper VAT-rate dropdown helper ($form->load_tva(...)) is present in core but commented out. The field is always a plain text box you type a number into (e.g. 10), with no GST/tax-code information.
  2. Even if a code-bearing value is submitted, it's discarded. Stock's save handler (action == 'save_price') parses tva_tx assuming it's a plain number, and both calls to $object->update_buyprice(...) hard-code an empty string for the VAT-code argument. So llx_product_fournisseur_price.default_vat_code is never set by this form.

Problem 2 is why KM2607 exists: products whose supplier price has tva_tx = 10 but default_vat_code IS NULL show no GST in the VAT dropdown on purchase order lines, because Dolibarr can't match a NULL code to a tax rule.

The fix we applied — three hunks, all interdependent

All three hunks work together. Hunk 3 (the dropdown) submits values like "10 (GST)"; hunk 1 must be able to parse that; hunk 2 must pass the parsed code through to be saved. Applying only one or two of these would either do nothing (hunk 2/3 without hunk 1 → "must be numeric" validation error on save) or silently drop the code again (hunk 3 without hunk 1+2).

Hunk 1 — parse the VAT code out of the submitted value (~line 217)

Stock 23.0.3Patched (current)
$npr = preg_match('/\*/', GETPOST('tva_tx', 'alpha')) ? 1 : 0;
$tva_tx = str_replace('*', '', GETPOST('tva_tx', 'alpha'));
if (!preg_match('/\((.*)\)/', $tva_tx)) {
    $tva_tx = price2num($tva_tx);
}
 
$npr = preg_match('/\*/', GETPOST('tva_tx', 'alpha')) ? 1 : 0;
$tva_tx = str_replace('*', '', GETPOST('tva_tx', 'alpha'));
$newdefaultvatcode = '';
if (preg_match('/\((.*)\)/', $tva_tx, $reg)) {
    $newdefaultvatcode = $reg[1];
    $tva_tx = price2num(preg_replace('/\s*\(.*\)/', '', $tva_tx));
} else {
    $tva_tx = price2num($tva_tx);
}

Stock's version only calls price2num() when there's no parenthesis — if a value like "10 (GST)" were ever submitted, $tva_tx would stay as that literal string, fail the later is_numeric($tva_tx) check, and the form would reject the save with "VAT rate must be numeric". This branch is effectively dead in stock because stock never produces a value with a code attached (no dropdown — see hunk 3). The patched version always extracts the code into $newdefaultvatcode and numeric-cleans $tva_tx either way.

Hunk 2 — pass the parsed code into update_buyprice() (~lines 333, 335)

Stock 23.0.3Patched (current)
// multicurrency
... array(), '', $multicurrency_price, ...

// non-multicurrency
... array(), '', 0, 'HT', 1, '', ...
 
// multicurrency
... array(), $newdefaultvatcode, $multicurrency_price, ...

// non-multicurrency
... array(), $newdefaultvatcode, 0, 'HT', 1, '', ...

Both calls to $object->update_buyprice(...) (multicurrency-enabled and disabled variants) take a VAT code as one of their later positional arguments. Stock hard-codes '' here, so default_vat_code is always cleared/blank regardless of what was selected. The patch passes $newdefaultvatcode from hunk 1 instead.

Why load_tva() was commented out in the first place

Per Dolibarr history, the dropdown was disabled by Laurent Destailleur (Dolibarr's original author), with the rationale that a supplier's VAT rate "may be any vat rates for any country" — i.e. for multi-country deployments, restricting the field to a dropdown of the local company's configured rates could be too limiting. The plain-text fallback avoided that restriction but, as a side effect, never wrote default_vat_code (hunks 1 & 2 above) — a bug that persisted unnoticed through Dolibarr v14–v23.

For our single-country (Australia-only) deployment, restricting the edit-price dropdown to locally configured GST rates is correct and desirable, so we've re-enabled it for the "edit existing price" case only — new prices for a potential overseas supplier still use free text.

Why this is safe

  • Scope is limited to the supplier-price edit form and its save handler — no other screens are touched.
  • Hunk 3 uses Dolibarr's own $form->load_tva() helper — the same function used for VAT dropdowns elsewhere in core.
  • Hunks 1–2 only change which value is written to the existing default_vat_code column — no schema changes.
  • The "add new price" path is untouched, so creating a brand-new supplier price still behaves exactly as stock.

⚠ Upgrade risk — this is a CORE file edit

htdocs/ is excluded from this repo's git tracking (see CLAUDE.md). All three hunks live only in the live htdocs/product/price_suppliers.php file on this machine and on the live server — they are not backed up by git.

  • Any future Dolibarr core upgrade (re-extracting/replacing htdocs/) will silently remove all three hunks and revert to stock behaviour (no dropdown, default_vat_code cleared on every save).
  • This page is the record of what to re-apply after an upgrade. Re-apply all three hunks together — applying a subset can produce a worse state than stock (e.g. hunk 3 alone causes a "must be numeric" save error).

Checked (2026-06-15): a formObjectOptions hook-based alternative is not viable for hunk 3. That hook (htdocs/product/price_suppliers.php around line 914) fires after the field table is printed and can only append rows, not replace the VAT field rendered earlier in the same table. Hunks 1–2 are in the save-handler, well before any hook point. A hook-based approach would still require core changes to call hooks at these new points, so it offers no upgrade-safety benefit over this patch.

How to test

  1. Go to Products > open a product with an existing supplier price > "For purchase" tab.
  2. Click the pencil/edit icon on an existing supplier price row (action=edit_price&rowid=NN).
  3. Confirm "Tax Rate (for this vendor/product)" is a dropdown, pre-selected to the product's current rate/code (e.g. 10% (GST)).
  4. Change it (or just re-save with the same value), click Save.
  5. Check the database row was written correctly: 
    SELECT rowid, fk_product, fk_soc, tva_tx, default_vat_code
FROM llx_product_fournisseur_price
WHERE rowid = NN;
    default_vat_code should be GST (or GST_FREE etc.), not NULL or empty.
  6. Click "Add price" (new supplier price, no rowid) and confirm the VAT field is still the original plain text box (unchanged).
  7. Add the new product/supplier combo to a purchase order line and confirm the VAT dropdown on the PO line shows GST 10% (not blank) — this is the symptom KM2607 was written for.

How to undo

In htdocs/product/price_suppliers.php, revert all three hunks back to the stock 23.0.3 versions shown in the left-hand columns above:

  • Hunk 1 (~line 217): restore the simple if (!preg_match(...)) { $tva_tx = price2num($tva_tx); } form, remove $newdefaultvatcode.
  • Hunk 2 (~lines 333, 335): change both $newdefaultvatcode arguments back to ''.
  • Hunk 3 (~line 647): remove the load_tva() call and $vatforselect/$default_vat_code logic, restore the commented-out line and the plain <input type="text"> for both branches.
